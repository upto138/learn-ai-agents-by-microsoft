## AI 에이전트 정의 및 유형

### AI 에이전트란?

간단히 말하면 이렇게 생각할 수 있습니다:

> **AI 에이전트는 대형 언어 모델(LLM)이 단순히 프롬프트에 응답하는 것을 넘어서, 도구와 지식을 활용해 실제로 <em>행동</em>할 수 있도록 만드는 시스템입니다.**

조금 더 풀어 설명하자면:

- <strong>시스템</strong> — AI 에이전트는 단일 구성 요소가 아닙니다. 여러 부분이 함께 작동하는 집합체입니다. 기본적으로 모든 에이전트는 세 가지 구성 요소를 갖습니다:
  - <strong>환경</strong> — 에이전트가 동작하는 공간입니다. 예를 들어 여행 예약 에이전트라면 이 환경은 예약 플랫폼 자체입니다.
  - <strong>센서</strong> — 에이전트가 현재 환경 상태를 파악하는 방법입니다. 여행 에이전트는 호텔 이용 가능 여부나 항공권 가격을 확인할 수 있습니다.
  - <strong>액추에이터</strong> — 에이전트가 행동을 취하는 방법입니다. 여행 에이전트는 방을 예약하거나 확인 메시지를 보내거나 예약을 취소할 수 있습니다.

![What Are AI Agents?](../../../translated_images/ko/what-are-ai-agents.1ec8c4d548af601a.webp)

- **대형 언어 모델** — 에이전트는 LLM 이전에도 존재했지만, LLM이 현대 에이전트를 강력하게 만든 핵심입니다. 자연어를 이해하고, 맥락을 추론하며, 모호한 사용자 요청을 구체적 실행 계획으로 변환할 수 있습니다.

- **행동 수행** — 에이전트 시스템이 없으면 LLM은 단지 텍스트를 생성할 뿐입니다. 에이전트 시스템 내에선 LLM이 실제로 단계를 <em>실행</em>할 수 있습니다 — 데이터베이스 검색, API 호출, 메시지 전송 등.

- **도구 접근** — 에이전트가 사용할 수 있는 도구는 (1) 실행되는 환경과 (2) 개발자가 부여한 기능에 따라 달라집니다. 여행 에이전트는 항공권 조회는 가능하지만 고객 기록을 수정할 순 없습니다 — 연결한 기능에 달렸습니다.

- **기억 + 지식** — 에이전트는 단기 기억(현재 대화)과 장기 기억(고객 데이터베이스, 과거 상호작용)을 지닐 수 있습니다. 여행 에이전트는 당신이 창가 좌석을 선호한다는 것을 "기억"할 수 있습니다.

---

### AI 에이전트의 다양한 유형

모든 에이전트가 같게 만들어지진 않습니다. 주요 유형을 여행 예약 에이전트를 예로 들어 설명합니다:

| **에이전트 유형** | <strong>역할</strong> | **여행 에이전트 예** |
|---|---|---|
| **단순 반사 에이전트** | 하드코딩된 규칙을 따름 — 기억도, 계획도 없음. | 불만 메일을 보면 → 고객 서비스로 전달함. 그게 전부. |
| **모델 기반 반사 에이전트** | 세상의 내부 모델을 유지하며 상황 변화를 반영함. | 과거 항공권 가격을 추적하고 갑자기 비싸진 노선을 표시함. |
| **목표 기반 에이전트** | 목표를 정하고 단계별로 달성 방법을 찾음. | 현재 위치에서 목적지까지 비행기, 차량, 호텔 예약 등 전체 여행을 예약함. |
| **효용 기반 에이전트** | 단순히 <em>해결</em>하는 게 아니라, 절충안을 고려해 <em>최적</em> 해결책을 찾음. | 비용과 편리함을 저울질해 사용자 선호에 가장 적합한 여행을 찾음. |
| **학습 에이전트** | 피드백을 통해 시간이 지날수록 더 나아짐. | 여행 후 설문 결과를 반영해 미래 예약 추천을 조정함. |
| **계층적 에이전트** | 상위 에이전트가 작업을 하위 에이전트에 위임해 세분화함. | "여행 취소" 요청이 비행기 취소, 호텔 취소, 차량 대여 취소 등으로 나뉘어 각각 하위 에이전트가 처리함. |
| **다중 에이전트 시스템 (MAS)** | 여러 독립 에이전트가 협력하거나 경쟁함. | 협력적: 각 에이전트가 호텔, 항공, 엔터테인먼트를 맡음. 경쟁적: 여러 에이전트가 호텔 객실 확보를 위해 경쟁함. |

## AI 에이전트를 사용해야 할 때

AI 에이전트를 <em>쓸 수 있다</em>고 해서 항상 *써야 하는 건 아닙니다*. 에이전트가 진가를 발휘하는 상황은 다음과 같습니다:

![When to use AI Agents?](../../../translated_images/ko/when-to-use-ai-agents.54becb3bed74a479.webp)

- **개방형 문제** — 문제 해결 단계가 미리 정해질 수 없을 때. LLM이 동적으로 경로를 찾아야 함.
- **다단계 프로세스** — 여러 턴에 걸쳐 도구를 사용해야 하는 작업, 단순 조회나 텍스트 생성이 아닌 경우.
- **시간에 따른 향상** — 사용자 피드백이나 환경 신호를 활용해 시스템이 점점 똑똑해져야 할 때.

이 과정의 뒤쪽 수업인 <strong>신뢰할 수 있는 AI 에이전트 구축하기</strong>에서 AI 에이전트를 써야 하는 경우와 *써서는 안 되는* 경우를 더 깊이 다룹니다.

## 에이전트 솔루션 기초

### 에이전트 개발

에이전트를 구축할 때 가장 먼저 해야 할 일은 *무엇을 할 수 있는지* 정의하는 것입니다 — 도구, 행동, 동작을 정합니다.

이 과정에서는 주요 플랫폼으로 <strong>Azure AI Agent Service</strong>를 사용합니다. 이 서비스는 다음을 지원합니다:

- OpenAI, Mistral, Llama 같은 개방형 모델
- Tripadvisor 같은 공급자 라이선스 데이터
- 표준화된 OpenAPI 3.0 도구 정의

### 에이전틱 패턴

LLM과의 소통은 프롬프트를 통해 이뤄집니다. 에이전트는 모든 프롬프트를 수동으로 작성할 수 없어서 여러 단계에 걸쳐 행동해야 합니다. 여기서 <strong>에이전틱 패턴</strong>이 등장합니다. 이는 LLM을 더 확장 가능하고 신뢰성 있게 프롬프팅하고 조율하기 위한 재사용 가능한 전략입니다.

이 과정은 가장 보편적이고 유용한 에이전틱 패턴을 중심으로 구성되어 있습니다.

### 에이전틱 프레임워크

에이전틱 프레임워크는 개발자에게 에이전트 구축에 필요한 템플릿, 도구, 인프라를 제공합니다. 덕분에 다음이 쉬워집니다:

- 도구와 기능 연결
- 에이전트 활동 관찰 및 문제 발생 시 디버깅
- 여러 에이전트 간 협업

이 과정에서는 생산 준비가 된 에이전트 구축을 위해 <strong>Microsoft Agent Framework (MAF)</strong>를 중점적으로 다룹니다.
