## AIエージェントの定義とエージェントの種類

### AIエージェントとは？

ざっくり言うと：

> **AIエージェントは、大規模言語モデル（LLM）にただの応答以上のことをさせるために、ツールや知識を与えて実際に行動できるようにするシステムです。**

詳しく見ていきましょう：

- <strong>システム</strong> — AIエージェントは単一のものではありません。複数の部品が連携して動く集合体です。基本的に全てのエージェントは三つの部分からなります：
  - <strong>環境</strong> — エージェントが働く空間。旅行予約エージェントなら予約プラットフォーム自体です。
  - <strong>センサー</strong> — エージェントが環境の現在の状態を読み取る方法。旅行エージェントはホテルの空き状況や航空券の価格をチェックします。
  - <strong>アクチュエーター</strong> — エージェントが行動を起こす方法。旅行エージェントは部屋の予約をしたり、確認メールを送ったり、キャンセルをしたりします。

![What Are AI Agents?](../../../translated_images/ja/what-are-ai-agents.1ec8c4d548af601a.webp)

- <strong>大規模言語モデル</strong> — エージェントはLLMがなくても存在していましたが、LLMがあることで現代のエージェントは非常に強力になりました。自然言語を理解し、文脈を考慮し、あいまいなユーザーの要求を具体的な行動計画に変えられます。

- <strong>行動の実行</strong> — エージェントシステムなしでは、LLMはテキストを生成するだけです。エージェントシステムの中では、LLMが実際に手順を実行できます。データベース検索、API呼び出し、メッセージ送信など。

- <strong>ツールへのアクセス</strong> — エージェントが利用できるツールは（1）動作している環境と（2）開発者が用意したものに依存します。旅行エージェントはフライトを検索できても顧客記録を編集できないかもしれません。接続されているもの次第です。

- **記憶＋知識** — エージェントは短期記憶（現在の会話）や長期記憶（顧客データベース、過去のやりとり）を持てます。旅行エージェントは「お客様が窓側の席を好む」ことを“覚えている”かもしれません。

### AIエージェントの種類

すべてのエージェントが同じ形で作られているわけではありません。以下は旅行予約エージェントを例にした主要なタイプの一覧です：

| <strong>エージェントの種類</strong> | <strong>特徴</strong> | <strong>旅行エージェントの例</strong> |
|---|---|---|
| <strong>単純反射エージェント</strong> | ルールに従うだけでメモリや計画なし。 | クレームメールを見たらカスタマーサービスに転送するだけ。 |
| <strong>モデルベース反射エージェント</strong> | 内部の世界モデルを保持し状況変化に応じて更新。 | 過去の航空券価格を追跡し、急に高騰したルートを警告する。 |
| <strong>目標志向エージェント</strong> | 目的を持ち、その達成方法を段階的に計画。 | 現在地から目的地へ、飛行機・レンタカー・ホテルすべて予約する。 |
| <strong>ユーティリティベースエージェント</strong> | 最適な解を選ぶためにトレードオフを考慮。 | コストと利便性をバランスさせて、あなたの嗜好に最適な旅行を提案。 |
| <strong>学習エージェント</strong> | フィードバックから学習し時間とともに改善。 | 旅行後のアンケート結果を元に未来の予約推奨を調整。 |
| <strong>階層型エージェント</strong> | 上位エージェントが作業を細分化し下位エージェントに委任。 | 「旅行キャンセル」リクエストをフライト、ホテル、レンタカーのキャンセルに分割。 |
| **マルチエージェントシステム（MAS）** | 複数の独立したエージェントが協働または競合。 | 協働ならホテル、飛行機、娯楽を別のエージェントが担当。競合なら複数エージェントが最安価格でホテルを埋めようと競う。 |

## AIエージェントを使うべき時

AIエージェントを使えるからといって、いつも使うべきとは限りません。以下の状況でエージェントは特に有効です：

![When to use AI Agents?](../../../translated_images/ja/when-to-use-ai-agents.54becb3bed74a479.webp)

- <strong>オープンエンドな問題</strong> — 解決手順があらかじめプログラムできない時。LLMが動的に道筋を考えます。
- <strong>複数段階のプロセス</strong> — 1回の検索や生成だけではなく、複数ターンにわたってツールを使う必要がある場合。
- <strong>時間経過による改善</strong> — 利用者のフィードバックや環境の変化に応じてシステムを賢くしたい時。

本コースの後半の<strong>信頼できるAIエージェント構築</strong>レッスンで、いつ使うべきか（使うべきでない時も）さらに掘り下げます。

## エージェントソリューションの基本

### エージェント開発

エージェントを作る際にまずすることは、「何ができるか」＝そのツールや行動、振る舞いを定義することです。

このコースでは主に **Azure AI Agent Service** を使います。対応しているのは：

- OpenAI、Mistral、Llamaなどのオープンモデル
- Tripadvisorなどのライセンス提供データ
- OpenAPI 3.0 標準ツール定義

### エージェントパターン

LLMとのやり取りはプロンプトで行いますが、エージェントでは手作業で全てのプロンプトを巧みに作るのは難しいです。エージェントは多段階の行動を実行する必要があるためです。そこで <strong>エージェントパターン</strong> が役立ちます。これは、LLMに効率よくかつ信頼できる形で指示を出し、オーケストレーションする再利用可能な戦略です。

このコースは最もよく使われる有用なエージェントパターンを軸に構成されています。

### エージェントフレームワーク

エージェントフレームワークは、開発者向けにテンプレートやツール、インフラを整えたものです。これにより、次のことが容易になります：

- ツールや機能の接続
- エージェントの動作監視（不具合発生時のデバッグ）
- 複数のエージェントでの協働

本コースでは **Microsoft Agent Framework (MAF)** を使い、本番利用に耐えうるエージェント構築を学びます。
